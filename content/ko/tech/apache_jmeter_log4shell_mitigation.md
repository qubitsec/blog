이 문서는 2021년 Log4Shell 대응 내용을 **2026년 기준으로 다시 정리한 문서**입니다.  
과거에는 `formatMsgNoLookups` 같은 임시 완화책이 널리 안내되었지만, 현재는 그것만으로 충분하다고 보기 어렵습니다.  
가장 중요한 원칙은 **가능한 한 빨리 Apache JMeter와 Log4j를 최신 안정 버전으로 올리는 것**입니다.

Apache JMeter는 2021년 12월 Log4Shell 대응 과정에서  
**5.4.2에서 Log4j 2.16.0**, **5.4.3에서 Log4j 2.17.0**을 반영했습니다.  
현재 JMeter 문서는 **5.6.3** 기준이며, 실행 환경은 Java 8 이상, **Java 17 이상 권장**입니다.

> 결론부터 말하면,  
> **최신 JMeter 버전 사용이 최우선**이며,  
> 개별 Log4j 설정만으로 문제를 해결하려는 접근은 이제 권장하기 어렵습니다.

## 왜 다시 정리해야 하나요?

2021년 12월 당시에는 Log4j **2.15.0 이상 업그레이드**나  
`-Dlog4j2.formatMsgNoLookups=true` 같은 임시 대응이 빠르게 공유되었습니다.

하지만 이후 추가 취약점과 우회 이슈가 이어졌고,  
지금 기준에서는 다음처럼 이해하는 것이 더 정확합니다.

- **2.15.0은 최종 안전 버전으로 보기 어렵습니다**
- `formatMsgNoLookups` 같은 옵션은 **임시 완화책**일 뿐, 장기 대응책이 아닙니다
- 가능하면 **JMeter 자체를 최신 버전으로 업그레이드**해야 합니다
- Log4j를 별도로 관리하는 경우에도 **최신 안정 버전 사용**이 원칙입니다

즉, 지금 이 주제는 “어떤 옵션을 넣을까”보다  
“**구버전을 계속 쓸 것인가, 최신 버전으로 정리할 것인가**”가 더 중요합니다.

---

## 현재 기준 권장 대응 순서

## 1) 가장 권장되는 방법: Apache JMeter 최신 버전 사용

가장 좋은 방법은 **Apache JMeter 자체를 최신 버전으로 업그레이드**하는 것입니다.

이유는 단순합니다.

- JMeter는 2021년 12월 긴급 대응 릴리스에서 이미 Log4j를 상향 조정했습니다
- 현재 JMeter 5.6.x 계열은 더 최신 의존성과 수정 사항을 반영합니다
- 개별 JAR를 수동 교체하는 방식보다 운영 실수가 적습니다

특히 신규 설치 또는 재설치가 가능하다면,  
과거 JMeter 디렉터리를 계속 보존하며 부분 수정하기보다  
**최신 배포본으로 교체 설치**하는 것이 더 안전합니다.

---

## 2) 수동 관리가 필요한 경우: Log4j도 최신 안정 버전 사용

조직 정책상 JMeter 내부 라이브러리를 직접 관리해야 한다면,  
Log4j도 **최신 안정 버전**으로 맞추는 것이 좋습니다.

과거처럼 “2.15.0 이상이면 된다”는 기준은 더 이상 적절하지 않습니다.  
현재 Apache Log4j는 2.25.x 계열 릴리스를 제공하고 있으며,  
JNDI Lookup도 2.17.0부터는 명시적으로 켜지지 않으면 활성화되지 않습니다.

즉, 수동 교체가 필요하다면 다음 원칙을 따르는 것이 좋습니다.

- 가능한 한 **최신 안정 Log4j 2.x** 사용
- 구버전 JAR 잔존 여부 확인
- 여러 위치의 중복 Log4j JAR 동시 존재 여부 확인
- 교체 후 실제 실행 classpath 기준으로 다시 점검

---

## 3) 업그레이드가 당장 어려운 경우: 임시 완화만 사용

구버전을 즉시 교체할 수 없는 환경에서는  
임시 완화 조치가 필요할 수 있습니다.

다만 여기서 중요한 점은,  
이 조치들은 **최종 해결책이 아니라 임시 대응**이라는 것입니다.

예전 문서에서 자주 안내되던 예시는 다음과 같습니다.

### Java 실행 인자 예시
```bash
-Dlog4j2.formatMsgNoLookups=true
```

### 환경 변수 예시

```bash
LOG4J_FORMAT_MSG_NO_LOOKUPS=true
```

### 설정 파일 예시

`PatternLayout`에서 `%m`을 `%m{nolookups}`로 변경

하지만 이런 방식은 2021년 초동 대응으로는 의미가 있었어도,  
현재는 **업그레이드를 미루기 위한 장기 대안으로 사용하면 안 됩니다.**

정리하면:

* 임시 완화는 가능
* 하지만 **최신 JMeter / 최신 Log4j 업그레이드가 우선**
* 완화 조치만 해두고 구버전을 장기간 유지하는 것은 권장하지 않음

---

## 4) 정말 불가피한 경우: JNDI 관련 기능 제거 검토

아주 오래된 환경이나 즉시 업그레이드가 불가능한 환경에서는  
JNDI 관련 클래스를 제거하거나 비활성화하는 방식이 검토될 수 있습니다.

다만 이 방법은 운영 환경마다 차이가 크고,  
수동 수정 과정에서 오히려 장애를 만들 수 있으므로  
현재 기준에서는 **기본 권장안이 아니라 예외적 대응책**으로 보는 것이 맞습니다.

즉, 우선순위는 아래와 같습니다.

1. **JMeter 최신 버전으로 업그레이드**
2. **Log4j 최신 안정 버전 사용**
3. 불가피할 때만 임시 완화
4. 아주 예외적인 경우에만 클래스 제거 등 강제 조치 검토

## JMeter 사용 시 추가로 확인할 점

JMeter는 주로 사용자 PC나 테스트 실행 환경에서 동작하는 도구입니다.  
그래서 “서버 프로그램이 아니니 덜 위험하다”라고 단순히 볼 수는 없습니다.

다음도 함께 점검해야 합니다.

* 테스트용 PC 또는 실행 서버에 **구버전 JMeter가 남아 있는지**
* 팀 내에서 ZIP으로 복사해 쓰는 **개별 사본이 여러 개 존재하는지**
* CI/CD 또는 성능 테스트 서버에 **오래된 JMeter 디렉터리**가 남아 있는지
* 플러그인이나 추가 라이브러리가 **별도 Log4j 의존성**을 가져오는지

즉, 단순히 “내 PC만 최신”이면 끝나는 문제가 아니라,  
**실제로 JMeter가 실행되는 모든 위치의 배포본과 라이브러리**를 확인해야 합니다.

## 실무 권고안

가장 실무적으로는 아래처럼 정리하는 편이 좋습니다.

### 권고 1

**새 설치 또는 재배포 시에는 항상 최신 JMeter 버전을 사용합니다.**

### 권고 2

기존 설치본은 `lib` 및 관련 경로의 Log4j JAR 버전을 점검하고,  
구버전이 남아 있으면 제거 또는 교체합니다.

### 권고 3

`formatMsgNoLookups` 같은 설정은  
업그레이드 전까지의 **한시적 완화책**으로만 사용합니다.

### 권고 4

정기적으로 의존성 점검 도구와 자산 점검을 수행합니다.

예:

* SBOM 점검
* SCA/의존성 취약점 점검
* 운영/테스트 도구 버전 인벤토리 관리

## 정리

2021년에는 빠른 완화가 중요했습니다.  
하지만 지금은 기준이 달라졌습니다.

이제는 다음처럼 이해하는 것이 맞습니다.

* **2.15.0 이상이면 충분하다** → 지금 기준에서는 부정확
* **워크어라운드만 적용하면 된다** → 지금 기준에서는 부족
* **최신 JMeter + 최신 Log4j 사용이 우선** → 현재 기준의 정답에 가까움

따라서 이 주제는 더 이상  
“JMeter에서 Log4Shell을 임시로 막는 방법”이 아니라,  
**“JMeter와 관련 라이브러리를 최신 상태로 유지하는 운영 가이드”**로 재정리하는 편이 좋습니다.
