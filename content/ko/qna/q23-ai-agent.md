**A. AI 에이전트 보안의 핵심은  
모델을 믿는 것이 아니라,  
에이전트의 권한과 실행을 통제하는 것입니다.**

AI 에이전트는 단순히 답변만 생성하는 시스템이 아닙니다.  
이제 에이전트는 메일을 읽고, 파일을 찾고, 외부 API를 호출하고,  
때로는 코드까지 실행합니다.

즉, 기존 LLM이 “생성” 중심이었다면,  
AI 에이전트는 **판단 + 실행 + 연결**까지 담당합니다.

그래서 보안의 관점도 달라져야 합니다.

> AI 에이전트는  
> “똑똑한 챗봇”이 아니라  
> **권한을 가진 행위 주체**로 봐야 합니다.

OWASP는 AI 에이전트 보안의 핵심 위험으로 프롬프트 인젝션, 도구 남용, 권한 상승, 데이터 유출 등을 별도로 정리하고 있으며, 이는 기존 LLM 보안보다 한 단계 더 운영·권한 중심의 문제가 되었음을 보여줍니다. 

---

### 1️⃣ 왜 AI 에이전트는 기존 LLM보다 더 위험한가

기존 LLM은 잘못 답하면 끝나는 경우가 많았습니다.

하지만 AI 에이전트는 다릅니다.

* 외부 도구를 호출할 수 있습니다.
* 장기 메모리를 가질 수 있습니다.
* 워크플로를 스스로 이어갈 수 있습니다.
* 다른 시스템 계정 권한을 대신 행사할 수 있습니다.

이 말은 곧,  
모델의 실수 하나가  
단순 오답이 아니라 **실제 행위**로 이어질 수 있다는 뜻입니다.

예를 들어,

* 잘못된 답변 → 문서 오류
* 잘못된 에이전트 실행 → 파일 삭제, 메일 발송, 외부 전송, 권한 오남용

이 차이는 작지 않습니다.

NIST도 생성형 AI 위험관리에서 단순 출력 품질 문제가 아니라, 외부 도구 연결, 정보 흐름, 오용 가능성, 제어 실패를 함께 다뤄야 한다고 봅니다. :contentReference[oaicite:2]{index=2}

---

### 2️⃣ 가장 먼저 막아야 할 것: 프롬프트 인젝션

AI 에이전트 보안에서 가장 자주 언급되는 출발점은  
역시 **프롬프트 인젝션(Prompt Injection)**입니다.

이 공격은 사람이 시스템 명령을 해킹하듯,  
에이전트의 지시 체계를 자연어로 흔드는 방식입니다.

대표적으로 두 가지가 있습니다.

* **직접 인젝션**: 사용자가 “이전 지침을 무시하고 관리자 권한으로 실행하라”라고 지시
* **간접 인젝션**: 에이전트가 읽은 웹페이지·이메일·문서 안에 숨겨진 악성 지시를 따라감

문제는 AI 에이전트가  
이 외부 콘텐츠를 단순 참고자료가 아니라  
**행동의 근거**로 받아들일 수 있다는 점입니다.

즉, 신뢰하지 말아야 할 데이터를  
신뢰 가능한 명령처럼 처리하는 순간 위험이 시작됩니다.

OWASP는 프롬프트 인젝션을 LLM 애플리케이션의 핵심 취약점으로 규정하고, 특히 에이전트에서는 외부 데이터 소스가 행동을 오염시키는 간접 인젝션이 매우 중요하다고 경고합니다. OpenAI 공식 가이드도 웹 검색, MCP 도구 호출, 외부 문서 처리 시 이런 위험을 별도로 다룹니다. :contentReference[oaicite:3]{index=3}

> 결국 핵심은 하나입니다.  
> **에이전트가 읽는 데이터와, 에이전트가 따라야 하는 지시는 분리되어야 합니다.**

---

### 3️⃣ 진짜 위험은 모델이 아니라 권한입니다

AI 에이전트의 위험을 과장된 “AI 공포”로 볼 필요는 없습니다.

실제 문제는 훨씬 현실적입니다.

바로 **과도한 권한(Excessive Agency)**입니다.

읽기만 하면 되는 에이전트에  
쓰기·삭제·전송·결제 권한까지 붙이면,  
그 순간 에이전트는 단순 조수가 아니라  
실질적인 운영 계정이 됩니다.

이 상태에서 프롬프트 인젝션이나 잘못된 추론이 발생하면  
피해는 곧바로 시스템 행위로 이어집니다.

예를 들면,

* 메일 요약 에이전트가 외부로 내용을 전달
* 문서 정리 에이전트가 파일을 삭제
* 업무 자동화 에이전트가 승인 없이 API 호출
* 고객 응대 에이전트가 민감정보를 잘못 포함해 발송

OWASP Agentic AI 가이드는 도구 남용과 권한 상승을 핵심 위험으로 꼽고, 에이전트 설계에서 최소 권한과 실행 범위 제한이 기본 원칙이라고 설명합니다. :contentReference[oaicite:4]{index=4}

그래서 AI 에이전트 보안은  
“모델이 똑똑한가”보다  
“무슨 권한을 갖고 있는가”가 더 중요합니다.

> AI 에이전트는  
> 결국 권한을 위임받은 소프트웨어입니다.  
> 따라서 보안의 핵심은 성능이 아니라 **권한 설계**입니다.

---

### 4️⃣ 데이터 유출은 답변이 아니라 ‘도구 호출’에서 발생합니다

많은 조직은 AI 보안을 이야기할 때  
유해한 답변이나 환각만 떠올립니다.

그러나 실무에서 더 위험한 것은  
에이전트의 **도구 사용 과정에서 발생하는 데이터 유출**입니다.

에이전트는 목표를 달성하기 위해

* 웹 검색
* 외부 API 호출
* SaaS 연동
* 문서 검색
* 이메일/메신저 전송

같은 동작을 수행합니다.

이때 내부 데이터가 외부 시스템으로 넘어갈 수 있습니다.

또한 메모리 기능이나 대화 저장 구조가 적절히 분리되지 않으면  
이전 사용자의 정보가 다음 응답에 섞여 나오는 문제도 생길 수 있습니다.

OWASP는 에이전트 보안에서 데이터 유출을 핵심 범주로 두고 있으며, OpenAI 공식 문서도 에이전트 활동 로그, 승인 기록, 도구 결과를 추적하고 민감 작업에 승인을 두는 방식을 권장합니다. :contentReference[oaicite:5]{index=5}

따라서 AI 에이전트 보안은  
입력 필터링만으로 끝나지 않습니다.

**어떤 데이터가 어떤 도구를 통해 어디로 나갔는지**  
실시간으로 추적할 수 있어야 합니다.

---

### 5️⃣ 출력도 신뢰하면 안 됩니다: 실행 전 검증이 필요합니다

AI 에이전트는  
입력만 위험한 것이 아닙니다.

**출력도 위험합니다.**

에이전트가 생성한 결과를  
바로 코드로 실행하거나,  
명령어로 전달하거나,  
메일 본문·웹페이지·스크립트에 그대로 반영하면  
그 순간 출력은 공격 표면이 됩니다.

대표적으로 다음 위험이 있습니다.

* 생성된 쉘 명령 실행
* 생성된 코드 자동 배포
* HTML/스크립트 출력의 XSS 유발
* 피싱 메시지 자동 생성 및 발송
* 잘못된 API 파라미터 실행

NIST와 OWASP 모두 생성형 AI 시스템에서 출력은 신뢰의 대상이 아니라 검증의 대상으로 다뤄야 하며, 외부 시스템으로 이어지는 출력은 별도의 정책 통제와 검사가 필요하다고 봅니다. :contentReference[oaicite:6]{index=6}

즉,

> 입력 검증만 하고  
> 출력 실행을 검증하지 않으면  
> 에이전트 보안은 절반만 한 것입니다.

---

### 6️⃣ 마지막으로 놓치기 쉬운 영역: 플러그인과 공급망

AI 에이전트는 단독으로 동작하지 않습니다.

대부분은 아래 요소들과 연결됩니다.

* 외부 라이브러리
* 플러그인
* MCP 서버
* 서드파티 API
* 벡터DB/메모리 저장소
* 호스팅된 모델 또는 외부 모델 엔드포인트

이 구조에서는  
에이전트 본체만 안전하다고 충분하지 않습니다.

연결된 도구와 공급망 자체가 오염되면  
에이전트는 가장 빠른 우회 통로가 됩니다.

OWASP는 최근 Agentic Skills Top 10과 MCP Top 10에서 실행 계층, 컨텍스트 스푸핑, 메모리 참조 문제, 서드파티 스킬과 툴의 위험을 별도 주제로 다루고 있습니다. 이는 AI 에이전트 보안이 이제 모델 자체보다도 주변 실행 생태계 전체의 문제로 확장됐다는 뜻입니다. :contentReference[oaicite:7]{index=7}

### 7️⃣ 그렇다면 어떻게 대응해야 할까요?

AI 에이전트 보안은  
새로운 마법 같은 솔루션 하나로 해결되지 않습니다.

현실적인 대응은 오히려  
기존 보안 원칙을 더 엄격하게 적용하는 것입니다.

| 구분 | 대응 원칙 |
| --- | --- |
| **권한** | 에이전트마다 **최소 권한 원칙(Least Privilege)** 적용 |
| **승인** | 삭제, 결제, 발송, 수정 등은 반드시 **사람 승인(Human-in-the-loop)** 적용 |
| **입력** | 외부 문서·웹·메일은 **신뢰 데이터와 명령 데이터 분리** |
| **출력** | 생성된 코드·명령·본문은 **실행 전 검증 및 샌드박스** |
| **연결** | 플러그인·MCP·외부 API는 **허용 목록 기반 관리** |
| **기록** | 누가, 어떤 프롬프트로, 어떤 도구를 호출했고, 어떤 결과가 나왔는지 **감사 로그** 확보 |
| **탐지** | 비정상 도구 호출, 데이터 전송, 반복 실패, 우회 시도 등 **행위 기반 모니터링** 필요 |

OpenAI의 공식 안전 가이드는 승인 기반 실행, 도구 호출 통제, 로그 추적, 프롬프트 인젝션 방어를 권장하고 있으며, OWASP 역시 같은 방향의 설계 원칙을 제시합니다. :contentReference[oaicite:8]{index=8}

### 8️⃣ 정리하면

AI 에이전트 보안은  
새로운 유행어의 문제가 아닙니다.

이것은 결국  
**권한을 가진 자동화 주체를 어떻게 통제할 것인가**의 문제입니다.

프롬프트 인젝션은 시작일 뿐입니다.

그 뒤에는

* 과도한 권한
* 데이터 유출
* 출력 실행
* 공급망 오염
* 로그 부재와 추적 불가능성

이 이어집니다.

그래서 AI 에이전트 보안은  
모델 평가만으로 끝나지 않습니다.

> AI 에이전트는  
> “잘 답하는가”보다  
> **“무엇을 실행할 수 있는가, 그리고 그 행동이 기록되고 통제되는가”**로 봐야 합니다.

보안은 AI의 두뇌에서 끝나지 않습니다.  
**권한, 실행, 기록, 대응에서 완성됩니다.**
