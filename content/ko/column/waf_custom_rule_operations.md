## 1️⃣ 웹방화벽은 왜 “관리”가 핵심인가

많은 조직이 웹방화벽을 다음과 같이 오해합니다.

* “WAF 설치했으니 웹 공격은 막고 있다”
* “제조사 시그니처 업데이트면 충분하다”
* “탐지만 하고 운영은 나중에 보자”

하지만 **현실의 공격은 이미 시그니처 기반 방어를 전제로 우회**합니다.

특히 최근 공격은 다음 특징을 가집니다.

* **기존 시그니처를 의도적으로 회피**
* 파라미터 구조·인코딩·순서 변경
* 정상 요청과 구분하기 어려운 **Low & Slow 공격**
* 🤖 **AI로 자동 생성되는 변종 페이로드**

👉 이 말은 곧,  
“**알려진 시그니처만 막는 WAF는 이미 과거의 보안**”이라는 뜻입니다.

---

## 2️⃣ 알려지지 않은 공격은 어떻게 탐지해야 하는가

시그니처 우회 공격을 막기 위한 핵심은 단 하나입니다.

> **실제 공격 트래픽을 관찰하고,  
> 그 결과를 즉시 ‘커스텀 룰’로 반영하는 것**

### 2-1. 탐지의 출발점은 “로그”

다음과 같은 로그가 쌓이고 있지 않다면,  
WAF 운영은 사실상 불가능합니다.

* 비정상 URI 접근 패턴
* 정상 응답(200)인데 **의심스러운 요청 흐름**
* 반복되는 파라미터 변형 시도
* 특정 시간대 집중되는 요청 폭증
* 차단은 안 됐지만 **사람이 보기엔 이상한 요청**

👉 **차단되지 않았다고 안전한 것이 아닙니다.**  
차단되지 않은 공격이 **다음 공격의 학습 데이터**가 됩니다.

---

## 3️⃣ 탐지 다음 단계는 반드시 “커스텀 룰”이다 🛡️

탐지만 하고 조치하지 않으면 의미가 없습니다.

### 3-1. 커스텀 룰이란 무엇인가

* 우리 서비스에만 의미 있는 **정책 기반 룰**
* 제조사 시그니처에 없는 **현실 공격 대응 룰**
* URI / 파라미터 / 메서드 / 빈도 / 세션 조합 기반

예를 들면:

* 특정 API에 `POST` 외 메서드 전면 차단
* 정상 사용 시 절대 등장하지 않는 파라미터 구조 차단
* 로그인 실패 패턴이 아닌 **행위 기반 차단**
* 업무 시간 외 관리자 페이지 접근 전면 차단

👉 이것이 **실제 방어력을 만드는 WAF 운영**입니다.

---

## 4️⃣ 이 작업은 “매일” 해야 한다 🔁

중요한 점은 빈도입니다.

> **커스텀 룰 업데이트는  
> 월 1회도, 분기 1회도 아닙니다.  
> 사실상 ‘매일’ 해야 합니다.**

### 이유는 명확합니다.

* AI로 공격 패턴 생성 비용은 **0에 수렴**
* 우회 공격 생성 속도는 **사람의 대응 속도를 초과**
* 어제 안전했던 룰이 **오늘은 무력화**

즉,

* **탐지 → 분석 → 룰 반영**  
* 이 사이클이 **자동화 또는 상시 운영**되지 않으면  
  WAF는 금세 무용지물이 됩니다.

---

## 5️⃣ 웹방화벽이 한 번 뚫리면, 해커가 유리해진다 ⚔️

웹방화벽이 무력화되는 순간,  
공격자는 **가장 유리한 고지**를 점령합니다.

* 인증 우회
* 세션 탈취
* 관리자 기능 탐색
* 내부 API 구조 파악
* 추가 공격을 위한 정찰 완료

특히 위험한 구간이 있습니다.

> **금요일 자정 ~ 월요일 아침**

* 운영 인력 최소
* 탐지 알림 확인 지연
* 대응 승인 체계 정지

👉 이 시간대에 WAF가 뚫리면  
**주말 내내 무방비 상태**가 됩니다.

---

## 6️⃣ 이렇게 운영하지 않으면, WAF는 존재 의미가 없다 ❌

다음 중 하나라도 해당된다면  
웹방화벽은 **공격 대응을 못 하는 상태**입니다.

* 시그니처 업데이트만 자동으로 돌리고 끝
* 탐지 로그를 사람이 보지 않음
* 커스텀 룰이 거의 없음
* “오탐 날까 봐” 룰 추가를 회피
* 공격 후에만 정책을 수정

👉 이 경우 WAF는  
**‘보안 장비’가 아니라 ‘구성된 장식물’에 가깝습니다.**

* AI로 공격을 만드는 시대에는  
  **운영하지 않는 WAF = 없는 WAF**
* **매일 커스텀 룰을 갱신하지 않으면**  
  웹방화벽은 공격을 막지 못한다
