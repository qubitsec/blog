## 1️⃣ 웹방화벽은 왜 “관리”가 핵심인가

많은 조직이 웹방화벽을 이렇게 오해합니다.

- “WAF 설치했으니 웹 공격은 막고 있다”
- “제조사 시그니처 업데이트면 충분하다”
- “탐지만 하고 운영은 나중에 해도 된다”

하지만 현실의 공격은 이미 **시그니처 기반 방어를 전제로 우회**합니다.

특히 최근 공격은 다음 특징을 가집니다.

- 기존 시그니처를 의도적으로 회피
- 파라미터 구조, 인코딩, 순서 변경
- 정상 요청과 구분하기 어려운 **Low & Slow 공격**
- 🤖 AI로 자동 생성되는 변종 페이로드
- 웹 페이지가 아니라 **API 엔드포인트**를 노리는 abuse
- 사람보다 **Bot**이 먼저 들어오는 로그인/회원가입/결제 공격

즉,  
**“알려진 시그니처만 막는 WAF는 이미 과거형 보안”** 입니다.

클라우드 WAF들도 이제 custom rules, rate limiting, bot 대응을 핵심 기능으로 내세우고 있습니다. 이는 “기본 룰셋만으로는 부족하다”는 시장 현실을 그대로 보여 줍니다. :contentReference[oaicite:2]{index=2}

## 2️⃣ 알려지지 않은 공격은 어떻게 탐지해야 하는가

시그니처 우회 공격을 막기 위한 핵심은 단 하나입니다.

> **실제 공격 트래픽을 관찰하고,  
> 그 결과를 즉시 커스텀 룰로 반영하는 것**

### 2-1. 탐지의 출발점은 “로그”

다음과 같은 로그가 쌓이지 않는다면  
WAF 운영은 사실상 불가능합니다.

- 비정상 URI 접근 패턴
- 정상 응답(200)인데 **의심스러운 요청 흐름**
- 반복되는 파라미터 변형 시도
- 특정 시간대에 집중되는 요청 폭증
- 차단은 안 됐지만 사람이 보기엔 이상한 요청
- 로그인/API 엔드포인트에 대한 과도한 호출
- Bot으로 의심되는 헤더, 세션, 호출 간격 패턴

👉 **차단되지 않았다고 안전한 것이 아닙니다.**  
차단되지 않은 공격은 **다음 룰을 만드는 학습 데이터**입니다.

---

## 3️⃣ 탐지 다음 단계는 반드시 “커스텀 룰”이다 🛡️

탐지만 하고 조치하지 않으면 의미가 없습니다.

### 3-1. 커스텀 룰이란 무엇인가

- 우리 서비스에만 의미 있는 **정책 기반 룰**
- 제조사 시그니처에 없는 **현실 공격 대응 룰**
- URI, 파라미터, 메서드, 빈도, 세션, 헤더 조합 기반

예를 들면,

- 특정 API에 `POST` 외 메서드 전면 차단
- 정상 사용 시 절대 등장하지 않는 파라미터 구조 차단
- 로그인 실패 횟수 자체가 아니라 **행위 조합 기반 차단**
- 업무 시간 외 관리자 페이지 접근 전면 차단
- 특정 API에 대한 rate limit 또는 challenge 적용

클라우드 WAF들도 custom rules와 rate limiting을 공식 핵심 기능으로 제공합니다. 결국 제품이 달라도 운영 원칙은 같습니다. **기본 룰셋 위에 우리 서비스 전용 정책을 덧입혀야 실제 방어력**이 생깁니다. :contentReference[oaicite:3]{index=3}

## 4️⃣ 2026년 기준 WAF 운영은 무엇이 달라졌나

예전에는 “ModSecurity + OWASP CRS 튜닝”만 알아도 실무에 꽤 가까웠습니다.  
지금은 조금 더 넓게 봐야 합니다.

### 4-1. OWASP CRS는 여전히 중요하지만, 최신 버전을 봐야 합니다
OWASP CRS는 여전히 대표적인 오픈소스 룰셋이며, 현재 4.x 계열이 최신입니다. CRS는 ModSecurity뿐 아니라 Coraza 같은 호환 엔진에서도 사용됩니다. :contentReference[oaicite:4]{index=4}

### 4-2. Coraza 같은 대안도 실무에서 의미가 커졌습니다
Coraza는 Go 기반 WAF로, ModSecurity SecLang과 CRS v4 호환을 강조합니다. 컨테이너, 프록시, 클라우드 네이티브 환경에서 관심이 커지는 이유가 여기에 있습니다. :contentReference[oaicite:5]{index=5}

### 4-3. 웹 보안은 이제 API Security와 Bot 대응까지 포함합니다
지금의 WAF 운영은 단순히 SQLi, XSS 차단만이 아닙니다.

- API 오용 탐지
- Rate limit
- Credential Stuffing 방어
- Bot / scraper 대응
- 관리자 URI 보호
- 파일 업로드 검증

즉, 커스텀 룰 운영은 더 이상 “예외적인 심화 작업”이 아니라  
**WAF 실무의 기본 작업**이 되었습니다. :contentReference[oaicite:6]{index=6}

## 5️⃣ ModSecurity / CRS 운영의 기본 원칙

### 5-1. 기본 룰셋을 먼저 신뢰하되, 그대로 두지는 마십시오
OWASP CRS는 강력한 출발점입니다.  
하지만 모든 서비스에 바로 맞지는 않습니다.

예를 들어 다음 전략이 현실적입니다.

- 먼저 **Detection Only** 또는 제한적 차단으로 시작
- 서비스별 오탐 패턴 수집
- 민감 엔드포인트부터 우선 튜닝
- 검증 후 점진적으로 차단 범위 확대

### 5-2. Paranoia Level은 “높을수록 좋다”가 아닙니다
CRS의 Paranoia Level(PL)은 탐지 강도를 높여 주지만, 그만큼 오탐 위험도 올라갑니다.

실무적으로는 보통 이런 접근이 현실적입니다.

- **PL1**: 기본 시작점
- **PL2**: 운영 가능한 보안 수준으로 자주 추천
- **PL3~PL4**: 고위험 자산, 제한된 경로, 충분한 튜닝 여력이 있을 때 신중 적용

즉, 전체 서비스에 일괄적으로 PL을 높이기보다  
**민감 URI / 관리자 경로 / API 일부에만 강화 정책**을 거는 편이 더 실용적입니다.

## 6️⃣ 실무에서 자주 쓰는 커스텀 룰 운영 패턴

### 6-1. 관리자 페이지 접근 제한
정상 사용자가 거의 없는 관리자 URI는 강하게 보호할 수 있습니다.

```apache
SecRule REQUEST_URI "@beginsWith /admin" \
    "id:100100,phase:1,deny,status:403,log,msg:'Admin path blocked outside allowlist',chain"
    SecRule REMOTE_ADDR "!@ipMatch 10.0.0.0/8,192.168.0.0/16"
```

### 6-2. 특정 API 메서드 제한

예를 들어 `/api/order`는 `POST`만 정상인데 `GET`, `PUT`, `DELETE`가 반복되면 이상 징후일 수 있습니다.

```apache
SecRule REQUEST_URI "@streq /api/order" \
    "id:100110,phase:1,deny,status:403,log,msg:'Unexpected method on order API',chain"
    SecRule REQUEST_METHOD "!@streq POST"
```

### 6-3. Rate limit 보강을 위한 전처리

WAF 자체 또는 상위 프록시와 연계해 로그인 시도 과다를 차단하는 방식은 매우 실용적입니다.

```apache
SecRule REQUEST_URI "@streq /login" \
    "id:100120,phase:1,pass,nolog,setvar:ip.login_counter=+1,expirevar:ip.login_counter=60"

SecRule IP:LOGIN_COUNTER "@gt 20" \
    "id:100121,phase:1,deny,status:429,log,msg:'Too many login attempts'"
```

### 6-4. 오탐이 잦은 파라미터 예외 처리

문제는 룰 추가보다 **예외를 어떻게 정확히 좁히느냐**입니다.

예를 들어 특정 파라미터가 정상적으로 특수문자를 많이 포함한다면,  
전체 룰을 끄는 대신 **해당 경로 + 해당 파라미터 + 특정 룰 ID만 제외**해야 합니다.

```apache
SecRule REQUEST_URI "@beginsWith /api/search" \
    "id:100130,phase:1,pass,nolog,ctl:ruleRemoveTargetById=942100;ARGS:query"
```

이런 식의 예외 처리가 중요합니다.  
**전체 차단 해제는 편하지만, 결국 공격면을 넓히는 지름길**이기 때문입니다.

## 7️⃣ 실제 오탐 튜닝은 어떻게 해야 하는가

이 부분이 가장 중요합니다.  
많은 조직이 WAF를 어렵게 느끼는 이유는  
“탐지는 되는데 오탐이 많다”는 경험 때문입니다.

### 7-1. 자주 나오는 오탐 사례

#### 사례 A. 검색 API의 특수문자 오탐

* 정상 사용자가 검색창에 특수문자를 입력
* SQLi/XSS 룰이 과민 반응
* 결과: 정상 검색 차단

**튜닝 원칙**  
전체 룰을 끄지 말고,

* 특정 경로
* 특정 파라미터
* 특정 룰 ID

로 범위를 좁혀 예외 처리합니다.

#### 사례 B. JSON API 바디 오탐

* JSON body 내부의 값이 룰셋에 걸림
* 모바일 앱이나 파트너 연동 API에서 자주 발생

**튜닝 원칙**

* `Content-Type`
* URI
* JSON 필드명
* 공격 규칙 ID

를 함께 보고 예외를 설계합니다.

#### 사례 C. 특정 User-Agent 또는 자동화 클라이언트

* 내부 배치나 파트너 시스템이 특이한 요청 패턴 사용
* 일반 브라우저 기준 룰셋에서는 이상하게 보일 수 있음

**튜닝 원칙**  
User-Agent만으로 예외 처리하지 말고,

* 고정 IP
* 인증 여부
* 특정 경로
* 시간대

같은 조건을 함께 묶는 편이 안전합니다.

## 8️⃣ 성능을 망치는 룰의 특징도 알아야 한다

좋은 룰은 차단만 잘하는 것이 아니라  
**서비스를 느리게 만들지 않는 룰**이어야 합니다.

### 주의할 점

* 과도한 정규표현식 중첩
* 모든 요청 바디에 무거운 검사 적용
* 불필요하게 긴 chain
* 관리자 URI처럼 좁은 범위에만 써야 할 룰을 전체 요청에 적용
* anomaly scoring 상승만 많고 실제 차단 기준이 모호한 구조

### 실무 원칙

* 민감 URI 우선 적용
* path 기반 선분기 후 정밀 검사
* 예외 처리 범위를 최대한 좁게 유지
* 룰 성능 테스트를 운영 전 수행
* 차단보다 먼저 **로그와 영향도** 확인

즉, WAF 운영은 보안만이 아니라  
**정확도와 성능의 균형**입니다.

## 9️⃣ 이 작업은 “매일” 해야 한다 🔁

중요한 점은 빈도입니다.

> **커스텀 룰 업데이트는
> 월 1회도, 분기 1회도 아닙니다.
> 사실상 ‘매일’ 해야 합니다.**

이유는 명확합니다.

* AI로 공격 패턴 생성 비용은 급격히 낮아졌고
* 우회 공격 생성 속도는 사람의 대응 속도를 앞서며
* 어제 안전했던 룰이 오늘은 무력화될 수 있기 때문입니다

실제로 CRS도 2026년에 multipart 관련 우회 취약점이 공개되며 수정되었습니다. 이 사례는 “기본 룰셋도 계속 검증·업데이트해야 한다”는 점을 보여 줍니다. ([GitHub][2])

즉,

* **탐지 → 분석 → 룰 반영**
* 이 사이클이 자동화되거나 상시 운영되지 않으면

WAF는 금세 무용지물이 됩니다.

## 🔟 웹방화벽이 한 번 뚫리면, 해커가 유리해진다 ⚔️

웹방화벽이 무력화되는 순간,  
공격자는 가장 유리한 고지에 올라섭니다.

* 인증 우회
* 세션 탈취
* 관리자 기능 탐색
* 내부 API 구조 파악
* 추가 공격을 위한 정찰 완료

특히 위험한 시간대가 있습니다.

> **금요일 자정 ~ 월요일 아침**

* 운영 인력 최소
* 탐지 알림 확인 지연
* 대응 승인 체계 정지

이 시간대에 WAF가 뚫리면  
**주말 내내 무방비 상태**가 될 수 있습니다.

## 1️⃣1️⃣ WAF 룰만으로 끝나지 않는다: 여기서 XDR이 필요해진다

WAF는 **입구**를 잘 봅니다.  
하지만 공격이 웹 계층을 넘어 서버 내부 행위로 이어지면,  
WAF만으로는 전체 사고를 설명하기 어렵습니다.

예를 들어,

* 웹 공격 시도 이후
* 서버에서 비정상 프로세스가 실행되고
* 계정 권한 사용이 바뀌며
* 외부 연결이 발생했다면

이 시점부터는 **로그 기반 행위 분석**이 필요합니다.

즉,

* **WAF 커스텀 룰**은 1차 차단
* **PLURA-XDR**은 이후 서버·계정·프로세스·외부 연결까지 연계 분석

이 조합이 중요합니다.

> **WAF는 들어오는 공격을 보고,  
> XDR은 들어온 뒤 무슨 일이 벌어졌는지를 봅니다.**

이렇게 해야

* 우회 공격이 실제로 성공했는지
* 내부 확산이 있었는지
* 추가 차단이 필요한지

까지 이어서 판단할 수 있습니다.

| 7  | 주말·야간 공격 시나리오 고려        | ✅     |
| 8  | WAF 이후 서버 행위까지 추적       | ✅     |
| 9  | “탐지 → 룰 → 재탐지” 반복 구조 유지 | ✅     |

## 1️⃣3️⃣ 결론

* 웹방화벽은 **설치로 끝나는 장비가 아닙니다**

  **XDR과의 연결**이 필요합니다
