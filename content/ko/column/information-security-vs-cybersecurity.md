🔐 정보보안은 중요합니다.  
문서도 보호해야 하고, 개인정보도 관리해야 하며, 계정 권한도 정리해야 합니다.  
보안 교육, 인증 대응, 백업 정책, 내부 규정도 모두 필요합니다.

하지만 여기서 반드시 구분해야 할 것이 있습니다.

**정보보안이라는 큰 이름 아래 모든 보안 업무를 한꺼번에 묶어 버리면,  
정작 공격자가 어떻게 침투하고 있는지를 다루는 사이버보안 논의는 흐려질 수 있습니다.**

문서 암호화, DRM, DLP, NAC, DB 암호화, 계정 권한 관리, 개인정보 보호, 인증 대응은 모두 정보보안의 중요한 구성요소입니다.  
그러나 이들은 대체로 공격자를 직접 추적하고 차단하는 영역이라기보다,  
**IT 통제와 정책 집행을 위한 애플리케이션**에 가깝습니다.

반면 사이버보안은 다릅니다.

사이버보안은 공격자가 실제로 들어오는 경로를 보고,  
웹 요청, 계정 로그인, 서버 이벤트, PC 행위 로그를 분석해  
침해를 탐지하고 차단하고 대응하는 일입니다.

이 구분이 없으면  
기업은 정보보안을 열심히 하고 있다고 생각하면서도  
정작 사이버 공격에는 제대로 대응하지 못할 수 있습니다.

![정보보안과 사이버보안의 차이](https://blog.plura.io/cdn/column/information-security-vs-cybersecurity.png)

## 1. 정보보안은 너무 넓은 말이 되었습니다

정보보안은 원래 중요한 개념입니다.

정보의 기밀성, 무결성, 가용성을 보호하는 전체 활동이기 때문입니다.  
기업과 기관이 정보를 안전하게 관리하기 위해 반드시 필요한 큰 틀입니다.

문제는 이 말이 현장에서 너무 넓게 사용된다는 점입니다.

예를 들어 다음과 같은 업무가 모두 정보보안으로 분류됩니다.

- 문서 암호화
- DRM
- DLP
- NAC
- DB 암호화
- DB 접근제어
- 데이터 마스킹
- 키 관리
- 계정 권한 관리
- 보안 교육
- 개인정보 보호
- 인증 대응
- 백업 정책
- 물리 보안
- 내부 규정

물론 이들은 모두 필요합니다.  
조직의 보안 수준을 유지하고, 내부 통제를 강화하고, 법과 인증 요구사항을 충족하는 데 중요한 역할을 합니다.

그러나 이 모든 것을 같은 수준에서 “사이버보안”이라고 부르기 시작하면 문제가 생깁니다.

정보보안 논의는 넓어지지만,  
사이버보안 논의는 오히려 얕아지기 때문입니다.

사이버보안은 단순히 정보를 안전하게 관리하는 일이 아닙니다.  
공격자가 실제로 어떤 경로로 들어오고 있는지,  
어떤 행위를 하고 있는지,  
어디서 차단해야 하는지를 보는 일입니다.

## 2. 정보보안 도구 중 상당수는 사이버보안이 아니라 IT 통제 애플리케이션에 가깝습니다

정보보안 영역에는 여러 도구와 업무가 포함됩니다.  
하지만 그 성격을 하나씩 보면 사이버보안과는 결이 다른 것들이 많습니다.

대표적인 항목을 살펴보면 다음과 같습니다.

| 항목 | 일반적 분류 | 실제 성격 | 사이버보안과의 차이 |
|---|---|---|---|
| 문서 암호화 | 정보보안 | 문서 사용 통제 애플리케이션 | 공격 탐지보다 문서 열람·저장·반출 통제 중심 |
| DRM | 정보보안 | 권한 기반 문서 관리 시스템 | 해킹 대응보다 문서 사용 정책 집행 중심 |
| DLP | 정보보안 | 데이터 반출 통제 시스템 | 침투 탐지보다 내부 정보 유출 방지 정책 중심 |
| NAC | 정보보안 | 네트워크 접속 통제 시스템 | 공격 분석보다 단말 접속 허용·차단 관리 중심 |
| DB 암호화 | 정보보안 | 저장 데이터 보호·규제 대응 시스템 | 공격 탐지보다 DB에 저장된 데이터의 암호화와 열람 통제 중심 |
| DB 접근제어 | 정보보안 | DB 접속·쿼리 권한 통제 시스템 | 침투 흐름 분석보다 사용자·관리자의 DB 접근 정책 집행 중심 |
| 데이터 마스킹 | 정보보안 | 개인정보·민감정보 노출 최소화 시스템 | 공격 차단보다 화면·개발·테스트 환경의 데이터 노출 통제 중심 |
| 키 관리 | 정보보안 | 암호키 생성·보관·사용 통제 시스템 | 공격 탐지보다 암호키 생명주기와 사용 권한 관리 중심 |
| 계정 권한 관리 | 정보보안 | IAM·권한 관리 시스템 | 공격 행위 탐지보다 계정 생성·권한·승인 관리 중심 |
| 보안 교육 | 정보보안 | 인식 제고·컴플라이언스 활동 | 실시간 공격 대응이 아니라 예방 교육 중심 |
| 개인정보 보호 | 정보보안 | 법·규정 준수 관리 | 해킹 탐지보다 개인정보 처리·보관·파기 기준 중심 |
| 인증 대응 | 정보보안 | 감사·증적·절차 관리 | 실제 공격 대응보다 인증 항목 충족 중심 |
| 백업 정책 | 정보보안·IT 운영 | 데이터 복구·운영 연속성 관리 | 공격 탐지보다 장애·랜섬웨어 이후 복구 중심 |
| 물리 보안 | 정보보안 | 출입·시설 통제 | 사이버 공격 행위 분석과는 별도 영역 |
| 내부 규정 | 정보보안 | 조직 정책과 절차 | 공격 차단 기술이 아니라 관리 기준 수립 |

이 표에서 중요한 것은  
이 항목들이 불필요하다는 뜻이 아닙니다.

오히려 모두 필요합니다.  
기업과 기관의 기본 보안 체계를 유지하기 위해 반드시 갖추어야 할 것들입니다.

다만 이들의 중심 목적은 대부분 다음에 가깝습니다.

> “사용자가 정책에 맞게 시스템을 사용하도록 통제하는 것”  
> “정보가 내부 규정에 맞게 관리되도록 하는 것”  
> “감사와 인증에 필요한 절차와 증적을 갖추는 것”  
> “업무 시스템과 사용자 환경을 관리하는 것”

즉, 이들은 대체로 **IT 통제와 정책 집행**의 성격이 강합니다.

반면 사이버보안의 중심 질문은 다릅니다.

> “공격자가 지금 어디로 들어오고 있는가?”  
> “웹 요청, 계정 로그인, 서버 이벤트, PC 행위 로그에서 침해 흔적이 보이는가?”  
> “탐지된 공격을 즉시 차단할 수 있는가?”  
> “공격 흐름을 재구성하고 대응할 수 있는가?”

이 차이를 구분하지 않으면  
사이버보안은 다시 문서 보안, 인증 대응, 내부 규정 관리로 희석됩니다.

### 추가로 구분해야 할 정보보안·IT 통제 영역

실무에서는 아래 항목들도 자주 정보보안으로 분류됩니다.  
그러나 대부분은 공격자의 침투 행위를 직접 추적하기보다,  
조직의 자산·권한·데이터·업무 환경을 통제하는 성격이 강합니다.

| 항목 | 실제 성격 | 사이버보안과 구분해야 할 이유 |
|---|---|---|
| PAM | 관리자 권한 관리 | 관리자 계정 사용을 통제하지만, 공격 전체 흐름 분석은 별도 필요 |
| SSO·MFA | 인증·접근 통제 | 계정 보호에는 중요하지만, 로그인 이후 행위 탐지는 별도 필요 |
| 비밀번호 정책 | 계정 정책 관리 | 계정 보안의 기본 통제이며 공격 행위 분석 자체는 아님 |
| 패치 관리 | 취약점 노출 축소 | 공격면을 줄이지만, 실제 공격 탐지·대응은 별도 필요 |
| 취약점 진단 | 위험 식별 | 취약점을 찾는 활동이지, 침해 행위 실시간 차단은 아님 |
| 보안 형상관리 | 설정 기준 준수 | 안전한 설정을 유지하지만, 공격자의 현재 행위 분석은 아님 |
| CSPM·SSPM | 클라우드·SaaS 설정 점검 | 설정 오류를 줄이는 통제이며, 침해 탐지와는 구분 필요 |
| MDM·EMM | 모바일 단말 관리 | 단말 등록·정책·분실 대응 중심이며 공격 상관분석은 별도 필요 |
| 매체제어·USB 제어 | 외부 저장매체 통제 | 데이터 반출 통제 성격이 강하며 침투 탐지는 별도 필요 |
| 출력물 보안·워터마크 | 문서 유출 억제 | 사용자 행위 억제와 책임 추적 중심 |
| 메일 아카이빙 | 증적 보관 | 감사·검색·보존 목적이며 피싱 탐지와는 구분 필요 |
| 로그 보관 | 감사 증적 관리 | 로그 저장 자체와 공격 분석은 다름 |
| GRC 시스템 | 정책·리스크·인증 관리 | 보안 거버넌스 도구이지 실시간 공격 대응 도구는 아님 |
| BCP·DR | 업무 연속성·재해 복구 | 사고 이후 복구 체계이며 침투 탐지와는 구분 필요 |

여기서 경계가 되는 영역도 있습니다.  
예를 들어 DB 접근제어, DB 감사, CASB, 이메일 보안, ZTNA, SASE 같은 도구는  
운영 방식에 따라 공격 탐지 기능을 일부 포함할 수 있습니다.  
하지만 단순히 접근 정책을 적용하고 로그를 보관하는 수준이라면  
여전히 사이버보안의 본질보다는 IT 통제에 가깝습니다.

따라서 기준은 제품 이름이 아니라 기능입니다.

> 공격자 행위를 실시간으로 탐지·차단·상관분석하는가?  
> 아니면 사용자, 권한, 데이터, 설정, 증적을 정책에 맞게 관리하는가?

이 질문으로 나누면  
정보보안과 사이버보안의 경계가 훨씬 선명해집니다.

## 3. 문서 암호화와 DB 암호화가 있다고 해서 웹 공격을 막을 수는 없습니다

문서 암호화는 문서 사용을 통제합니다.  
DRM은 문서 권한을 관리합니다.  
DLP는 데이터 반출을 제한합니다.  
NAC는 네트워크 접속을 관리합니다.  
DB 암호화는 저장 데이터를 보호합니다.  
DB 접근제어는 데이터베이스 접속 권한을 통제합니다.  
데이터 마스킹은 개인정보와 민감정보의 노출을 줄입니다.  
계정 권한 관리는 사용자의 권한을 정리합니다.

모두 중요한 정보보안 활동입니다.

하지만 이 자체가 곧 사이버 공격 대응은 아닙니다.

문서 암호화가 있다고 해서  
웹셸 업로드 공격을 탐지할 수 있는 것은 아닙니다.

DLP가 있다고 해서  
SQL Injection 공격을 실시간으로 분석할 수 있는 것은 아닙니다.

NAC가 있다고 해서  
크리덴셜 스터핑 공격을 계정 행위 흐름으로 추적할 수 있는 것은 아닙니다.

DB 암호화가 있다고 해서  
SQL Injection 공격이나 웹셸 업로드 공격을 실시간으로 탐지할 수 있는 것은 아닙니다.

DB 접근제어가 있다고 해서  
공격자가 애플리케이션 계정을 탈취해 수행한 비정상 쿼리의 전체 침투 흐름이 자동으로 재구성되는 것은 아닙니다.

데이터 마스킹이 있다고 해서  
공격자의 내부 이동이나 서버 명령 실행을 탐지할 수 있는 것은 아닙니다.

백업 정책이 있다고 해서  
랜섬웨어가 어떤 경로로 들어왔는지 알 수 있는 것도 아닙니다.

계정 권한 관리가 있다고 해서  
공격자가 탈취한 계정으로 로그인한 뒤 어떤 행위를 했는지 자동으로 파악되는 것도 아닙니다.

따라서 정보보안 도구와 사이버보안 도구는 구분해서 봐야 합니다.

정보보안 도구는 주로 **사용자, 정보, 접근 권한, 저장 데이터를 통제**합니다.  
사이버보안 도구는 **공격자와 공격 행위**를 봅니다.

이 둘을 구분하지 않으면  
기업은 보안 솔루션을 많이 도입하고도  
정작 공격자의 침투 흐름은 보지 못하는 상황에 놓일 수 있습니다.

---

## 4. “공격은 웹만 있는 것이 아니다”라는 말은 맞지만, 논점을 흐리면 안 됩니다

사이버보안 논의에서 자주 발생하는 문제가 있습니다.

웹 공격을 이야기하면 곧바로 이런 반론이 나옵니다.

> “공격은 웹만 있는 것이 아닙니다.”  
> “PC 악성코드도 있습니다.”  
> “이메일 피싱도 있습니다.”  
> “내부자 위협도 있습니다.”

맞는 말입니다.

공격은 웹에만 존재하지 않습니다.  
PC 악성코드도 있고, 이메일 피싱도 있고, 계정 탈취도 있고, 내부자 위협도 있습니다.

하지만 이 말이 웹 공격 대응 논의를 흐리는 방식으로 사용되어서는 안 됩니다.

웹 공격을 말하는 것은  
“웹만 보자”는 뜻이 아닙니다.

외부에 공개된 웹사이트, API, 로그인 페이지, 관리자 페이지, 파일 업로드 기능, 검색 기능, 게시판, 결제 페이지가  
공격자에게 가장 먼저 노출되는 대표적인 공격면이라는 점을 말하는 것입니다.

따라서 웹 공격을 논의할 때는  
논점을 다른 곳으로 옮기기보다 먼저 다음을 물어야 합니다.

- 우리 웹 요청 원문을 보고 있는가?
- 공격자가 보낸 요청 본문과 파라미터를 분석하고 있는가?
- 로그인 실패와 성공의 흐름을 함께 보고 있는가?
- 크리덴셜 스터핑 공격을 계정 행위로 추적하고 있는가?
- 웹 공격 IP가 이후 서버나 계정 이벤트와 연결되는가?
- 웹 공격 이후 호스트에서 이상 행위가 발생했는가?

이 질문에 답하지 못한다면  
사이버보안의 핵심을 놓치고 있는 것입니다.

---

## 5. 사이버보안의 본질은 공격 흐름을 보는 것입니다

사이버보안은 제품 이름의 문제가 아닙니다.  
사이버보안은 공격 흐름을 볼 수 있느냐의 문제입니다.

공격자는 보통 하나의 지점에서 끝나지 않습니다.

웹 취약점을 찌르고,  
계정을 탈취하고,  
서버에 파일을 업로드하고,  
명령을 실행하고,  
권한을 올리고,  
내부로 이동하고,  
데이터를 빼내려 합니다.

따라서 사이버보안은 다음 흐름을 연결해서 봐야 합니다.

> **웹 공격 → 계정 행위 → 서버 이벤트 → PC 행위 → 데이터 유출 시도 → 대응**

이 흐름을 보지 못하면  
각 보안 제품은 따로 움직이게 됩니다.

WAF는 웹 공격만 보고,  
EDR은 PC나 서버 행위만 보고,  
SIEM은 로그를 모으기만 하고,  
DLP는 반출만 보고,  
인증 대응은 체크리스트만 보게 됩니다.

그러면 전체 공격 흐름은 보이지 않습니다.

사이버보안의 핵심은  
각각의 이벤트를 따로 보는 것이 아니라  
공격자의 행위를 하나의 흐름으로 재구성하는 것입니다.

---

## 6. 좋은 사이버보안은 로그를 중심으로 연결되어야 합니다

사이버보안에서 가장 중요한 것은 로그입니다.

웹 요청 로그,  
계정 로그인 로그,  
서버 이벤트 로그,  
PC 행위 로그,  
보안 장비 로그,  
애플리케이션 로그가 연결되어야 합니다.

로그가 없으면 공격은 보이지 않습니다.  
로그가 흩어져 있으면 공격 흐름은 끊어집니다.  
로그를 모으기만 하고 분석하지 않으면 대응은 늦어집니다.

그래서 사이버보안은 다음 질문에 답할 수 있어야 합니다.

- 누가 접속했는가?
- 어디에서 접속했는가?
- 어떤 요청을 보냈는가?
- 어떤 계정으로 로그인했는가?
- 어떤 프로세스가 실행되었는가?
- 어떤 파일이 생성되었는가?
- 어떤 권한 변화가 있었는가?
- 어떤 외부 통신이 발생했는가?
- 어떤 데이터가 유출되려 했는가?

이 질문에 답할 수 있어야  
공격을 탐지하고, 차단하고, 분석하고, 재발을 막을 수 있습니다.

즉, 사이버보안은 단순한 정책 관리가 아닙니다.  
**실제 공격 흔적을 로그로 보고 대응하는 체계**입니다.

---

## 7. 이런 관점에서 PLURA-XDR 같은 통합 보안 플랫폼이 의미를 갖습니다

이 지점에서 PLURA-XDR 같은 통합 보안 플랫폼이 의미를 갖습니다.

중요한 것은 특정 제품 이름 자체가 아닙니다.  
핵심은 무엇을 가능하게 하느냐입니다.

사이버보안이 제대로 작동하려면  
웹, 서버, PC, 계정, 보안 장비의 이벤트가 하나의 흐름으로 연결되어야 합니다.

PLURA-XDR은 이런 관점에서 다음과 같은 방향을 지향합니다.

- **WAF**로 웹 요청과 공격 시도를 분석
- **EDR**로 서버와 PC의 행위 이벤트를 분석
- **SIEM/XDR**로 여러 로그를 상관 분석
- **포렌식**으로 사고 원인과 범위를 확인
- **자동 대응**으로 공격 확산을 빠르게 차단

즉, 단순히 보안 제품을 많이 도입하는 것이 아니라  
공격 흐름을 하나로 연결해 보는 것이 중요합니다.

정보보안 도구가 사용자를 통제하고 규정을 집행하는 데 초점을 둔다면,  
사이버보안 플랫폼은 공격자의 행위를 추적하고 차단하는 데 초점을 두어야 합니다.

---

## 8. 이제 질문을 바꿔야 합니다

정보보안 관점에서는 이런 질문을 합니다.

- 문서가 암호화되어 있는가?
- 사용자의 권한이 적절한가?
- 개인정보 처리 절차가 있는가?
- 인증 항목을 충족했는가?
- 백업 정책이 있는가?
- 내부 규정이 마련되어 있는가?

모두 필요한 질문입니다.

하지만 사이버보안 관점에서는 질문이 달라져야 합니다.

- 공격자가 지금 어떤 경로로 들어오고 있는가?
- 웹 요청 원문을 분석하고 있는가?
- 계정 탈취 시도를 탐지하고 있는가?
- 서버와 PC에서 공격 행위를 추적하고 있는가?
- 공격 IP와 계정, 프로세스, 파일, 네트워크 이벤트를 연결하고 있는가?
- 탐지 후 즉시 차단할 수 있는가?
- 사고 이후 원인과 범위를 빠르게 설명할 수 있는가?

이 질문에 답할 수 있어야  
기업은 진짜 사이버보안을 하고 있다고 말할 수 있습니다.

## ✍️ 결론

**정보보안은 필요합니다.  
하지만 정보보안이 곧 사이버보안은 아닙니다.**

문서 암호화, DRM, DLP, NAC, 계정 권한 관리, 보안 교육, 개인정보 보호, 인증 대응, 백업 정책, 물리 보안, 내부 규정은 모두 중요합니다.  
그러나 이들은 대부분 IT 통제와 정책 집행에 가까운 영역입니다.

사이버보안은 다릅니다.

사이버보안은 공격자가 실제로 들어오는 경로를 보고,  
웹 요청과 계정 로그인, 서버 이벤트와 PC 행위 로그를 연결해,  
공격을 탐지하고 차단하고 대응하는 실전 방어 영역입니다.

그래서 기업은 다음을 분명히 구분해야 합니다.

### 1. 정보보안은 큰 틀이다
정보와 업무 체계를 안전하게 관리하는 전체 활동입니다.

### 2. IT 통제는 필요하지만 사이버보안의 본질은 아니다
문서 통제, 권한 관리, 인증 대응은 중요하지만 공격 흐름을 직접 보는 것은 아닙니다.

### 3. 사이버보안은 공격자 행위를 보는 일이다
웹, 계정, 서버, PC, 네트워크, 데이터 유출 시도를 로그로 연결해 탐지·차단·대응해야 합니다.

> 정보보안이라는 이름으로 사이버보안을 흐리면 안 됩니다.  
> 사이버보안은 별도로 깊게 다루어야 할 실전 공격 대응 영역입니다.

결국 중요한 것은  
“보안 항목을 갖추었는가”가 아니라  
**“공격자가 들어왔을 때 볼 수 있고, 막을 수 있고, 설명할 수 있는가”**입니다.

그 질문에 답하는 것이  
진짜 사이버보안의 출발점입니다.
