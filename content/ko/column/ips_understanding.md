🛡️ **침입차단시스템(IPS)**은 더 이상 “패킷을 보고 차단하는 장비 하나”로 설명하기 어렵습니다.  
오늘날 공격은 **암호화 트래픽**, **웹 애플리케이션**, **정상 도구 악용(LOLBAS/LOTL)**, **클라우드와 원격접속**, **호스트 내부 행위**까지 확장되었습니다.

그래서 IPS를 제대로 이해하려면,  
단일 네트워크 장비의 기능이 아니라 **어디에서 위협을 보고, 어디에서 차단하며, 어디에서 증적을 남기는가**의 관점으로 다시 봐야 합니다.

과거에는 네트워크 경계에서 패킷을 차단하는 것이 중심이었습니다.  
하지만 지금은 그 경계가 흐려졌습니다.  
**암호화된 트래픽은 네트워크 중간에서 잘 보이지 않고**,  
실제 공격은 **웹 본문, 계정 행위, 프로세스 실행, 파일 생성, 내부 확산**처럼 더 깊은 곳에서 드러납니다.

따라서 현대의 IPS는 다음을 함께 봐야 합니다.

* **네트워크 레벨 차단**
* **응용 계층 분석**
* **호스트 내부 행위 탐지·차단**
* **로그 상관분석과 실시간 대응**

즉, IPS는 단일 기능이 아니라  
**침입을 사전에 줄이고, 진행 중인 공격을 막고, 사후에 설명 가능한 증적을 남기는 통합 방어 체계**로 이해해야 합니다.

## 1. IPS를 다시 봐야 하는 이유

IPS라는 용어는 오랫동안 “침입을 탐지하고 차단하는 시스템”으로 사용되어 왔습니다.  
이 정의 자체는 틀리지 않지만, 지금은 너무 좁습니다.

예전 IPS는 주로 다음에 강했습니다.

* IP, 포트, 프로토콜 기반의 비정상 트래픽 차단
* 시그니처 기반 공격 패턴 탐지
* 네트워크 경계에서의 인라인 차단

하지만 지금 공격 환경은 달라졌습니다.

* 웹 공격은 대부분 **HTTPS/TLS** 안으로 들어옵니다.
* 정상 프로세스와 정상 도구를 악용하는 공격이 늘었습니다.
* 사용자는 SaaS, VPN, 원격근무, 클라우드 환경으로 분산되었습니다.
* 공격자는 네트워크보다 **애플리케이션과 계정, 호스트 내부 행위**를 더 많이 노립니다.

그래서 이제 중요한 질문은  
“IPS 장비가 있는가?”가 아니라,  
**“우리 조직은 어디에서 공격을 실제로 보고, 실제로 차단할 수 있는가?”** 입니다.

## 2. OSI 모델 기반으로 본 IPS의 실제 역할

네트워크 보안을 설명할 때 OSI 모델은 여전히 유용합니다.  
다만 각 계층을 따로따로 보는 것이 아니라, **계층별로 보이는 것과 안 보이는 것**을 구분해야 합니다.

### Layer 3: 네트워크 계층

여기서는 주로 **IP 주소, 라우팅, 네트워크 경로** 관점의 통제가 이뤄집니다.

**대표 대응**
* 비인가 IP 차단
* IP 스푸핑 탐지
* 네트워크 세그먼트 간 접근 제한
* 비정상 트래픽 경로 차단

**대표 공격 예시**
* 스푸핑
* 비정상 내부망 접근
* 우회 경로를 통한 스캔
* 잘못 설계된 라우팅 정책 악용

Layer 3에서의 IPS는 공격 내용을 깊게 이해하기보다,  
**어디에서 어디로 가는가**를 기준으로 위험을 줄이는 역할을 합니다.

### Layer 4: 전송 계층

여기서는 **포트, 세션, TCP/UDP 연결 상태**를 봅니다.

**대표 대응**
* 포트 기반 차단
* 비정상 세션 수 제한
* SYN Flood, 비정상 연결 패턴 완화
* 상태 기반 검사(Stateful Inspection)

**대표 공격 예시**
* 포트 스캔
* SYN Flood
* 대량 연결 시도
* 비정상 세션 지속

Layer 4는 오늘날에도 여전히 중요합니다.  
특히 대량 트래픽 기반 공격이나 기본적인 공격 표면 축소에는 효과적입니다.  
다만 이 계층만으로는 **정상 세션 안에 숨어 들어온 공격**을 구분하기 어렵습니다.

### Layer 7: 응용 계층

현대 보안의 핵심은 결국 여기에 있습니다.  
실제 비즈니스 로직과 데이터가 존재하는 곳이기 때문입니다.

**대표 대응**
* HTTP 요청 및 응답 분석
* 웹 본문(POST-body) 기반 탐지
* SQL Injection, XSS, 파일 업로드 공격 차단
* API 오남용 탐지
* 인증 우회, 세션 오남용 탐지

**대표 공격 예시**
* SQL Injection
* XSS
* 웹셸 업로드
* 경로 우회(Path Traversal)
* 인증 우회
* 대량 크리덴셜 스터핑

Layer 7에서 가장 중요한 점은,  
**공격은 정상 요청처럼 보일 수 있다**는 것입니다.  
즉, IP와 포트만 봐서는 알 수 없고,  
실제 요청 본문과 응답 흐름, 세션 맥락까지 봐야 합니다.

## 3. 네트워크 기반 IPS: 방화벽, NGFW, WAF

네트워크 기반 IPS는 여전히 중요합니다.  
다만 그 역할과 한계를 정확히 이해해야 합니다.

### 3.1 전통적 방화벽(Firewall)

전통적 방화벽은 기본적으로  
**IP, 포트, 프로토콜**을 기준으로 허용과 차단을 수행합니다.

이 방식은 지금도 필수입니다.

* 불필요한 포트 차단
* 내부망 직접 접근 통제
* 네트워크 분리
* 기본 공격 표면 축소

하지만 한계도 분명합니다.

* HTTPS 내부의 실제 요청 내용은 보지 못함
* 정상 포트(예: 443)로 들어온 공격은 잘 드러나지 않음
* 애플리케이션과 사용자 맥락이 부족함

즉, 전통적 방화벽은 **첫 번째 문**으로는 훌륭하지만,  
그 자체가 모든 침입차단을 대표할 수는 없습니다.

### 3.2 차세대 방화벽(NGFW)

최근에는 IPS를 설명할 때 NGFW를 빼기 어렵습니다.  
NGFW는 기존 방화벽보다 더 나아가 **애플리케이션 인식, 심층 패킷 검사, 위협 방지 기능**을 통합하는 방향으로 발전했습니다.

이런 진화는 분명 의미가 있습니다.

* 단순 포트 기반이 아니라 애플리케이션 기준 통제
* 사용자와 서비스 맥락 반영
* IPS 기능과 정책 통합
* 더 세밀한 인라인 보안 제어

하지만 여기서도 중요한 현실은 같습니다.

**암호화가 강화될수록, 네트워크 중간에서 모든 내용을 계속 보는 것은 점점 더 어려워진다**는 점입니다.

그래서 NGFW는 강력한 도구이지만,  
그 자체가 완전한 답은 아닙니다.

### 3.3 웹방화벽(WAF)

WAF는 응용 계층, 특히 웹 공격 대응에 특화된 IPS로 이해할 수 있습니다.

**장점**
* HTTP/HTTPS 요청 분석
* 웹 공격 패턴 탐지
* 업로드, 인증, 관리자 기능 같은 고위험 URI 통제
* 정책 기반 세밀한 차단

**한계**
* TLS 종료 및 복호화 설계가 필요
* 웹 외 프로토콜 공격에는 한계
* 애플리케이션 구조를 이해하지 못하면 오탐·미탐 가능

그럼에도 웹 공격 비중이 높은 현실에서는  
WAF는 현대적 IPS의 핵심 축 중 하나입니다.

## 4. 암호화 시대의 IPS 한계와 전환점

오늘날 IPS를 어렵게 만드는 가장 큰 변화 중 하나는 **암호화 트래픽의 보편화**입니다.

예전에는 네트워크 중간에서 패킷 내용을 보고 차단하는 방식이 더 잘 통했습니다.  
하지만 지금은 TLS 1.3 환경에서 가시성 확보가 훨씬 더 까다로워졌습니다.  
즉, 네트워크 IPS만으로는 공격의 실질 내용을 보기 어려운 상황이 늘고 있습니다.

이 때문에 현대의 IPS는 반드시 다음 질문으로 넘어가야 합니다.

* 복호화는 어디에서 일어나는가
* 그 지점에서 무엇을 볼 수 있는가
* 네트워크에서 안 보이는 것은 호스트에서 어떻게 볼 것인가
* 차단 후 증적은 어떻게 남길 것인가

결국 보안은 다시 **복호화 이후의 지점**,  
즉 **웹 서버, 애플리케이션, 호스트 내부**로 이동합니다.

---

## 5. 호스트 기반 IPS: 이제 더 중요해진 이유

네트워크에서 보이지 않는 것은 결국 호스트에서 드러납니다.

공격자는 HTTPS 안으로 들어와도,  
최종적으로는 서버나 PC 안에서 다음과 같은 행동을 해야 합니다.

* 프로세스를 실행한다
* 파일을 쓴다
* 권한을 올린다
* 계정을 만든다
* 외부와 다시 통신한다
* 내부로 확산한다

즉, **침해의 실체는 호스트 내부 행위**로 나타납니다.

### 대표적인 호스트 기반 보안 구성

1. **HIPS**
   * 호스트 내부의 이상 행위를 직접 감시하고 차단
2. **EDR**
   * 프로세스, 파일, 메모리, 네트워크 행위를 추적하고 대응
3. **애플리케이션 제어**
   * 허용된 실행만 통과시키는 방식
4. **감사 로그 및 Sysmon**
   * Windows 행위를 구조화된 이벤트로 남김
5. **Linux auditd / eBPF 기반 가시성**
   * execve, 파일 쓰기, 권한 상승, 네트워크 행위 추적

특히 Linux 환경에서는 eBPF가 관찰성과 보안 가시성을 확장하는 기술로 부상했습니다.  
다만 eBPF도 만능은 아니며, 기본 원칙은 여전히 같습니다.

**프로세스, 파일, 권한, 네트워크를 남겨야 한다.**

즉, 호스트 기반 IPS의 본질은  
“엔드포인트 안에서 실제 공격 행위를 잡아내는 것”입니다.

## 6. IPS와 EDR, SIEM, DLP의 관계를 어떻게 봐야 하는가

IPS를 설명하다 보면 흔히  
“그럼 EDR은 IPS인가?”,  
“SIEM도 IPS인가?” 같은 질문이 나옵니다.

정답은 단순 분류보다 **역할 중심**으로 이해하는 것이 낫습니다.

### EDR
EDR은 호스트 수준에서 침입 행위를 탐지하고 대응합니다.  
따라서 현대적 의미에서는 **호스트 기반 IPS의 핵심 축**으로 볼 수 있습니다.

### SIEM
SIEM은 직접 차단하는 도구라기보다,  
여러 로그를 모아 **상관분석과 가시성**을 제공합니다.  
즉, IPS의 두뇌 역할에 가깝습니다.

### DLP
DLP는 데이터 유출을 막는 데 초점이 있습니다.  
직접적인 침입차단이라기보다, **침입 이후의 피해 확산 차단**에 가깝습니다.

### WAF / NGFW / EDR / SIEM의 관계
이들을 따로따로 나열하면 복잡해 보이지만,  
실제로는 이렇게 이해하면 쉽습니다.

* **NGFW / Firewall**: 네트워크 진입 통제
* **WAF**: 웹 공격 차단
* **EDR / HIPS**: 호스트 행위 차단
* **SIEM / XDR**: 상관분석과 대응 orchestration
* **DLP**: 유출 억제

즉, IPS는 더 이상 특정 제품군 이름이 아니라  
**침입을 어디에서 어떻게 끊을 것인가에 대한 설계 철학**에 가깝습니다.

## 7. 2026년 기준으로 IPS를 이해할 때 꼭 추가해야 할 것들

### 7.1 Zero Trust 관점

Zero Trust는 보안을 넓은 네트워크 경계에서 좁은 자산과 사용자, 리소스 중심으로 이동시키는 관점입니다.

이 관점은 IPS에도 중요합니다.

예전에는  
“내부망이면 신뢰”라는 가정이 있었습니다.

지금은 다릅니다.

* 사용자 검증
* 디바이스 상태 검증
* 세션 맥락 검증
* 최소 권한
* 자산 단위 정책

즉, IPS는 더 이상 단순한 네트워크 차단 장비가 아니라  
**접근 맥락과 행위 맥락을 함께 보는 통제 체계** 안에서 이해해야 합니다.

### 7.2 클라우드와 원격접속

IPS는 더 이상 데이터센터 경계에만 있지 않습니다.

* 퍼블릭 클라우드
* 쿠버네티스
* 원격근무
* SaaS
* API 중심 서비스

이 환경에서는 전통적 네트워크 경계만으로는 한계가 분명합니다.  
그래서 IPS는 점점 더 **클라우드 워크로드**, **웹 게이트웨이**, **호스트 에이전트**, **로그 상관분석**과 결합합니다.

### 7.3 AI 기반 이상징후 탐지

AI나 ML 기반 anomaly detection이 점점 더 많이 언급되지만,  
여기서도 중요한 것은 모델보다 **입력 데이터**입니다.

네트워크 패킷 몇 개만 가지고 AI가 모든 것을 해결해 주지는 않습니다.

오히려 실무에서는 다음이 더 중요합니다.

* 웹 요청·응답 본문
* 프로세스 실행 이력
* 파일 생성·변경
* 계정 행위
* 네트워크 연결 이력
* 위협 인텔리전스와 내부 로그의 상관분석

즉, AI는 IPS를 대체하는 마법이 아니라,  
**좋은 로그가 있을 때 IPS를 더 똑똑하게 만드는 보조 수단**입니다.

---

## 8. PLURA 관점에서 다시 정의하는 IPS

Plura 블로그에서 IPS를 다시 설명한다면,  
핵심은 명확합니다.

**IPS는 네트워크 경계 장비의 이름이 아니라, 실제 침입을 보고 막고 설명할 수 있는 통합 체계**라는 점입니다.

이 관점에서 보면 PLURA-XDR의 역할도 자연스럽게 보입니다.

### 왜 통합 관점이 중요한가

네트워크 보안만으로는 부족합니다.  
호스트만 봐도 부족합니다.  
로그만 모아도 부족합니다.

중요한 것은 이 셋이 연결되는가입니다.

* 웹에서 어떤 요청이 들어왔는가
* 그 요청 이후 서버에서 어떤 프로세스가 실행되었는가
* 파일이 생성되었는가
* 계정이나 권한 변화가 있었는가
* 외부 반출 시도가 있었는가

이 흐름이 연결되어야  
비로소 “탐지”가 아니라 **실제 침입차단**이 됩니다.

### PLURA-XDR이 들어가는 지점

* **웹 계층**
  * 웹 요청과 응답, 특히 본문 기반 가시성 확보
* **호스트 계층**
  * 행위 기반 탐지와 차단
* **로그 계층**
  * 웹·OS·계정·파일·네트워크 이벤트 상관분석
* **대응 계층**
  * 차단, 격리, 포렌식 연결

즉, Plura 관점의 IPS는  
**WAF + EDR + 로그 상관분석 + 실시간 대응**을 끊어진 장비들의 집합이 아니라  
**하나의 운영 체계**로 보는 접근에 가깝습니다.

---

## 9. IPS의 포괄적 정의

이제 IPS를 다시 정의해 보면 다음과 같습니다.

### 전통적 정의
> IPS는 네트워크 트래픽을 검사해 악성 행위를 탐지하고 차단하는 시스템이다.

이 정의는 틀리지 않지만, 지금은 부족합니다.

### 현대적 정의
> IPS는 네트워크, 응용계층, 호스트, 로그 분석을 결합해  
> 침입을 사전에 줄이고, 진행 중인 공격을 차단하며,  
> 사후에 설명 가능한 증적까지 확보하는 통합 보안 체계다.

이 정의가 중요한 이유는 분명합니다.

* 네트워크만으로는 HTTPS 안의 공격을 다 보기 어렵고
* 호스트만으로는 최초 진입 맥락을 놓칠 수 있으며
* 로그만으로는 실시간 차단이 늦을 수 있기 때문입니다

따라서 오늘날 IPS는 단일 장비가 아니라  
**네트워크 + 웹 + 호스트 + 로그 + 대응**의 조합으로 이해해야 합니다.

---

## 10. 결론

IPS를 오래된 의미로만 보면,  
이제는 시대에 맞지 않습니다.

지금의 IPS는 단순한 네트워크 패킷 차단 장비가 아니라,

* **경계에서 줄이고**
* **응용계층에서 걸러내고**
* **호스트 내부에서 막고**
* **로그를 연결해 설명하고**
* **필요하면 실시간 대응까지 이어지는**

**통합 보안 전략**이어야 합니다.

특히 암호화와 클라우드, 원격접속, 정상 도구 악용이 일상화된 환경에서는  
“네트워크 IPS 하나 있으면 된다”는 생각이 가장 위험합니다.

이제 더 정확한 표현은 이것입니다.

> **현대의 IPS는 장비가 아니라 체계입니다.**  
> **패킷 차단만이 아니라, 웹·호스트·로그를 연결해 실제 침입을 끊는 운영 모델**입니다.

그리고 이 관점에서 보면,  
PLURA-XDR 같은 통합형 보안 플랫폼은  
IPS를 오래된 장비 개념에서 꺼내  
**실제 탐지·차단·포렌식이 이어지는 운영 체계**로 바꾸는 방향에 더 가깝습니다.
